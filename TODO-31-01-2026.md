## Structure de projet (Décisions)

### Philosophie adoptée: **Option 1 - Minimal**

> **Knot = compilateur R↔Typst, PAS orchestrateur de documents**

**Principes:**
- Knot ne gère PAS la structure Typst (dossiers, images, bibliographie)
- L'utilisateur utilise le système de projet Typst natif
- Knot fournit juste les helpers (knot.typ, knot.R)

### Template minimal

**Structure:**
```
my-project/
├── knot.toml           # Configuration Knot
├── main.knot           # Document principal
└── lib/
    ├── knot.typ        # Helpers Typst (copie locale)
    └── knot.R          # Helpers R (copie locale)
```

**knot.toml:**
```toml
[document]
main = "main.knot"

[helpers]
typst = "lib/knot.typ"
r = "lib/knot.R"
```

**Pourquoi copie locale ("poorman"):**
- Pas encore publié sur CRAN/Typst Universe
- Permet développement indépendant
- Transition facile vers packages publiés plus tard:
  - `typst = "@preview/knot"` (futur)
  - `r = "@cran/knot"` (futur)

### Distribution template

**Stratégie: A + C (Hybride)**
- **A:** Embarqué dans le binaire (via include_dir!)
- **C:** Override possible dans `~/.knot/templates/`

**Commande:**
```bash
knot init my-project         # Crée le projet avec template minimal
```

### Architecture multi-fichiers (Future - Livres/Documents longs)

**Cas d'usage:** Rédaction de livres avec un chapitre par fichier.

**Principe de design:** Isolation R par fichier pour reproductibilité et parallélisation.

#### Structure projet livre

```
my-book/
├── .gitignore                  # Généré par knot init
├── knot.toml
├── main.knot                   # Structure du livre (source, éditable)
├── .main.typ                   # Généré (caché - dot file)
├── 01-intro.knot               # Chapitre 1 (source)
├── .01-intro.typ               # Généré (caché)
├── 02-methods.knot             # Chapitre 2 (source)
├── .02-methods.typ             # Généré (caché)
├── 03-results.knot             # Chapitre 3 (source)
├── .03-results.typ             # Généré (caché)
├── figures/                    # Figures sources (versionnées)
│   └── diagram.svg
├── generated-r-figures/        # Plots R (gitignored, visible pour debug)
│   └── plot-abc123.svg
├── generated-python-figures/   # Plots Python (futur, gitignored)
├── generated-lilypond-figures/ # Partitions LilyPond (futur, gitignored)
├── lib/
│   ├── knot.typ
│   └── knot.R
└── .knot_cache/                # Caché
```

**Convention dot files:**
- Fichiers `.typ` générés sont préfixés avec `.` (cachés)
- Invisibles dans `ls` par défaut
- Visibles avec `ls -la` pour debug
- Convention Unix pour fichiers temporaires/générés

**Organisation figures:**
- `figures/` → Sources manuelles (versionnées dans git)
- `generated-r-figures/` → Plots générés par R (gitignored)
- `generated-python-figures/` → Plots Python futur (gitignored)
- `generated-lilypond-figures/` → Partitions LilyPond futur (gitignored)

**.gitignore généré:**
```gitignore
# Knot generated files
.*.typ

# Generated figures by language
generated-*-figures/

# Knot cache
.knot_cache/

# PDF output
*.pdf
```

#### knot.toml étendu

```toml
[document]
main = "main.knot"
chapters = [
  "01-intro.knot",
  "02-methods.knot",
  "03-results.knot"
]

[helpers]
typst = "lib/knot.typ"
r = "lib/knot.R"
```

#### main.knot avec placeholder

**Template minimal pour multi-fichiers:**
```typst
#import "lib/knot.typ": *
#import "@preview/charged-ieee": ieee

#show: ieee.with(
  title: [Machine Learning for Genomics],
  authors: [Nicolas Klutchnikoff]
)

/* KNOT-INJECT-CHAPTERS */
```

**Placeholder:**
- Format: `/* KNOT-INJECT-CHAPTERS */` (commentaire Typst, pas de chevrons)
- Syntaxiquement valide pour le LSP (tinymist ne se plaint pas)
- Détectable et remplaçable par regex simple

**Traitement par `knot build`:**
- Détecte le commentaire `/* KNOT-INJECT-CHAPTERS */`
- Remplace par les includes Typst (fichiers générés avec dot):
```typst
#include ".01-intro.typ"
#include ".02-methods.typ"
#include ".03-results.typ"
```
- Génère `.main.typ` (dot file caché)

**Compatibilité single-file:** Si pas de placeholder, reste un fichier normal.

#### Commandes de compilation

**Compilation incrémentale (édition):**
```bash
knot compile 02-methods.knot  # Compile 1 seul fichier
```
- Rapide pour l'édition
- Typst recompile incrémentalement le PDF

**Compilation complète (build):**
```bash
knot build  # Compile tous les fichiers depuis knot.toml
```
- Lit knot.toml
- Compile main.knot → main.typ (avec substitution placeholder)
- Compile chaque chapitre → .typ correspondant
- Chaque fichier a son **propre RExecutor isolé**

#### Isolation R par fichier

**Principe:**
```
01-intro.knot   → RExecutor #1 → 01-intro.typ
02-methods.knot → RExecutor #2 → 02-methods.typ
03-results.knot → RExecutor #3 → 03-results.typ
```

**Bénéfices:**
- ✅ **Reproductible:** Pas de dépendances cachées entre chapitres
- ✅ **Parallélisable:** Compilation simultanée possible
- ✅ **Debuggable:** Erreurs isolées par chapitre
- ✅ **Scalable:** Gros livres compilent vite

**Limitation acceptée:**
- Variables R non partagées entre chapitres
- Code commun via `source("lib/utils.R")` (bonne pratique)

#### Cache par fichier

```
.knot_cache/
├── main/
├── 01-intro/
│   ├── chunk-123.csv
│   └── chunk-456.svg
├── 02-methods/
│   └── chunk-789.svg
└── 03-results/
    └── chunk-abc.csv
```

**Isolation complète, pas de collision.**

#### Avantages de l'architecture

1. **Pas d'import .knot dans .knot** → LSP reste simple
2. **Utilise #include natif de Typst** → Pas de réinvention
3. **Orchestration via knot.toml** → Configuration claire
4. **Compatible templates Typst** → N'importe quel template fonctionne
5. **Parallélisation** → Performance pour gros projets

**Philosophie:** "Knot = compilateur, pas orchestrateur"
